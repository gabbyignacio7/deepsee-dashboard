# DeepSee AI Dashboard - Tooltip Enhancement (React/TypeScript)

## OBJECTIVE
Add informative hover tooltips to the DeepSee prioritization dashboard to help Sales team, new employees, and executives understand the five key priority framework metrics without needing to ask Product/Engineering for definitions.

## CONTEXT
**Dashboard Technology**: React/TypeScript (Next.js framework)
**Current Deployment**: Replit - https://deepsee-dashboard-v1.replit.app
**Target Audience**: 
- Steve (Sales/CEO) - needs simple explanations for client conversations
- New team members - need onboarding context
- Board members - need high-level understanding without technical jargon

**Why This Matters**: 
Ryan specifically requested these tooltips after feedback that stakeholders were unclear on terminology during dashboard reviews. The priority framework uses specific formulas, and tooltips will reduce meeting interruptions for definitions and enable self-service understanding.

### 2. ARR (Annual Recurring Revenue)
**Icon to Use**: üíµ or üìà

**Tooltip Text**:
```
ARR (Annual Recurring Revenue)

What it is: The total annual contract value from a client for this feature or opportunity. This is the actual or projected recurring revenue per year.

Example:
‚Ä¢ Client signs 3-year contract for $750K total
‚Ä¢ ARR = $250K/year

Why it matters: ARR is the primary revenue metric we use to measure business value and prioritize features that drive sustainable, recurring income.

Note: This is the "maximum potential" - see Weighted ARR for probability-adjusted values.
```

**Technical Notes**:
- Field name in data: `ARR_Amount`
- Data type: Currency (USD)
- Display format: `$XXX,XXX` with comma separators
- Used in: All views, Priority Score calculations, Revenue Impact analysis

---

### 3. REPLICABILITY SCORE
**Icon to Use**: üîÑ or üåê

**Tooltip Text**:
```
Replicability Score

What it is: How many clients would benefit from this feature. Higher scores mean the feature can be reused across multiple clients (platform feature), while lower scores indicate client-specific customization.

Scale: 1-10
‚Ä¢ 10 = Benefits ALL clients (universal platform feature)
‚Ä¢ 7-9 = Benefits multiple clients (high reusability)
‚Ä¢ 4-6 = Benefits some clients (moderate reusability)
‚Ä¢ 1-3 = Benefits one client only (custom work)

Why it matters: Platform features with high replicability create more long-term value than one-off customizations. We prioritize work that can serve multiple clients.

Example: 
‚Ä¢ Email automation agent = 10 (all clients use email)
‚Ä¢ Custom DTCC integration = 2 (specific to one client)
```

**Technical Notes**:
- Field name in data: `Replicability_Score`
- Data type: Integer (1-10)
- Display format: Whole number with "/10" suffix (e.g., "8/10")
- Used in: Priority Score calculations, Platform vs. Custom analysis

---

### 4. CONVERSION PROBABILITY
**Icon to Use**: üéØ or üìä

**Tooltip Text**:
```
Conversion Probability

What it is: The likelihood that a sales opportunity will close successfully and become a signed contract, expressed as a percentage.

Scale: 0-100%
‚Ä¢ 90-100% = Contract signed or imminent
‚Ä¢ 70-89% = Strong interest, likely to close
‚Ä¢ 50-69% = Qualified opportunity, moderate confidence
‚Ä¢ 25-49% = Early stage, uncertain outcome
‚Ä¢ 0-24% = Long shot or exploratory

Why it matters: Used to calculate Weighted ARR and helps us focus on opportunities most likely to generate revenue. Higher probability deals get higher priority.

Example:
‚Ä¢ POC completed successfully = 80% probability
‚Ä¢ Initial discovery call only = 30% probability
```

**Technical Notes**:
- Field name in data: `Conversion_Probability_%` or `Conversion_Probability`
- Data type: Percentage (0-100)
- Display format: `XX%` (whole number percentage)
- Used in: Weighted ARR calculations, Sales Pipeline forecasting

### 5. EFFORT (T-SHIRT SIZE)
**Icon to Use**: ‚è±Ô∏è or üèóÔ∏è

**Tooltip Text**:
```
Effort (T-Shirt Size)

What it is: Estimated engineering effort required to deliver the feature, categorized by complexity and time investment.

Size Categories:
‚Ä¢ XS (Extra Small) = 1-3 days
  Simple changes, minor UI updates, quick fixes

‚Ä¢ S (Small) = 4-7 days (1 week)
  Small features, isolated changes, single component work

‚Ä¢ M (Medium) = 2-3 weeks
  Standard feature development, multiple components

‚Ä¢ L (Large) = 4-6 weeks
  Complex features, significant architecture changes

‚Ä¢ XL (Extra Large) = 7+ weeks (2+ months)
  Major platform changes, new agent development

Why it matters: Helps balance workload and identify "quick wins" (low effort, high value). Used in priority calculations to favor features with better value-to-effort ratios.

Priority Formula: (ARR √ó Replicability √ó Conversion Probability) √∑ Effort
```

**Technical Notes**:
- Field name in data: `Effort_T-Shirt_Size` or `effort_tshirt_size`
- Data type: Categorical (XS, S, M, L, XL)
- Display format: Text with size abbreviation
- Numeric conversion for calculations: XS=1, S=2, M=3, L=5, XL=8 (Fibonacci-like scale)
- Used in: Priority Score calculations, Sprint planning, Capacity planning

## TECHNICAL IMPLEMENTATION GUIDE

### TOOLTIP COMPONENT REQUIREMENTS

#### Option 1: Custom Tooltip Component (Recommended)
```typescript
// components/Tooltip.tsx
import React, { useState } from 'react';

interface TooltipProps {
  icon?: string;
  title?: string;
  content: string;
  maxWidth?: string;
  children?: React.ReactNode;
}

export const Tooltip: React.FC<TooltipProps> = ({ 
  icon, 
  title, 
  content, 
  maxWidth = '400px',
  children 
}) => {
  const [isVisible, setIsVisible] = useState(false);

  return (
    <div 
      className="tooltip-container"
      onMouseEnter={() => setIsVisible(true)}
      onMouseLeave={() => setIsVisible(false)}
    >
      {children || (
        <span className="tooltip-trigger">
          {icon || '‚ÑπÔ∏è'}
        </span>
      )}
      
      {isVisible && (
        <div 
          className="tooltip-content"
          style={{ maxWidth }}
        >
          {title && <div className="tooltip-title">{title}</div>}
          <div className="tooltip-body">{content}</div>
        </div>
      )}
    </div>
  );
};
```

## ACCESSIBILITY REQUIREMENTS

### Keyboard Navigation
```typescript
// Ensure tooltips are keyboard accessible
const Tooltip = ({ content, children }) => {
  const [isVisible, setIsVisible] = useState(false);

  return (
    <span
      role="tooltip"
      tabIndex={0}
      onMouseEnter={() => setIsVisible(true)}
      onMouseLeave={() => setIsVisible(false)}
      onFocus={() => setIsVisible(true)}
      onBlur={() => setIsVisible(false)}
      aria-label={content}
    >
      {children}
    </span>
  );
};
```

### ARIA Labels
- All tooltip triggers should have `aria-label` or `aria-describedby`
- Tooltip content should have `role="tooltip"`
- Ensure screen readers can access tooltip content

### Mobile Considerations
- On mobile, tooltips should appear on tap, not hover
- Provide a close button for mobile tooltips
- Ensure tooltip text is readable on small screens (min 12px font)

```typescript
// Mobile-friendly tooltip behavior
const isMobile = /iPhone|iPad|iPod|Android/i.test(navigator.userAgent);

const handleTooltipInteraction = (e) => {
  if (isMobile) {
    e.preventDefault();
    setIsVisible(!isVisible);
  }
};
```
